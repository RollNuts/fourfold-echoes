# Project Art Audit

Date: 2026-06-25

This audit records observed facts for the private `fourfold-echoes` Unity
project. It intentionally separates confirmed facts from assumptions and
unknowns so later art work is not pulled back into discarded prototype choices.

## Confirmed

| Area | Finding | Evidence |
| --- | --- | --- |
| Unity version | Unity 6.3 LTS `6000.3.18f1` | `ProjectSettings/ProjectVersion.txt` |
| Render pipeline | Built-in render pipeline | `ProjectSettings/GraphicsSettings.asset` has `m_CustomRenderPipeline: {fileID: 0}` and quality custom pipelines are `{fileID: 0}` |
| Canonical evidence scene | D-020 scene/evidence generation is the current target | scene generation lands in a separate Veripsa lane |
| Current visual state | No final production art is approved | D-020 art direction and asset register |
| Generated Unity assets | Generated scene assets remain evidence only until their lane lands and passes visual review | Art lanes must not treat generated primitives as final production art |
| Blender availability | Blender CLI runs locally and generated pilot outputs | `blender --background --python tools/Blender/generate_pilot_assets.py` |
| Blender version observed | Blender `5.1.2` | `artifacts/Reports/blender-pilot-assets.json` |
| Blender pilot source | `Assets/Art/Generated/BlenderPilot/Source/FE_BlenderPilotAssets.blend` | Generated by `tools/Blender/generate_pilot_assets.py` |
| Blender pilot exports | Hero, boss, and ruin kit FBX files exist | `Assets/Art/Generated/BlenderPilot/Models/` |
| Blender pilot previews | Hero, boss, and ruin kit preview PNGs exist | `artifacts/Previews/BlenderPilot/` |
| Asset pipeline validation | Blender pilot manifest validates generated files and public-safe relative paths | `node tools/AssetPipeline/validate_generated_assets.mjs` |
| Unity import | Unity imported the BlenderPilot folders and FBX files during `Scripts/Validation/run_all.sh` | Unity batchmode import log from 2026-06-25 |
| License status | Current pilot assets are repository-authored | `docs/ASSET_RIGHTS.md` |

## Assumed

| Area | Assumption | Reason |
| --- | --- | --- |
| Unit scale | 1 Unity unit equals 1 meter | Unity default and current C# generation code use meter-like dimensions |
| Camera language | Fixed-angle camera remains the near-term review camera | Current product-review evidence and validation capture use fixed camera |
| Art scope | Pilot models and generated scene assets remain pilot-only | No production hero, enemy, boss, or environment model is approved |
| Region and tool variants | Most location/tool differences should use shared geometry plus material/emission/decal changes | Current art bible and asset register require scalable production |

## Unknown

| Area | Unknown |
| --- | --- |
| Final protagonist shape | No protagonist model has been approved by the user |
| Final boss shape | No production boss model has been approved by the user |
| Final animation rig | Humanoid vs Generic is not selected for production characters |
| Final material pipeline | Built-in pipeline is current, but production shader/material conventions are still pilot-level |
| Controller/gameplay scene wiring | Production gameplay prefabs are not established |
| NavMesh/collider acceptance | Generated BlenderPilot assets do not yet have Unity collider validation |

## Immediate Art Pipeline Implication

The next large step is not more placeholder composition. It is the Hero Area:

1. Replace the pilot hero and boss with production-intent silhouettes.
2. Build one compact landmark-led Hero Area rather than a room-only proof.
3. Capture the required hero/tool/enemy/room/chest turnaround and gameplay images.
4. Judge whether the screen reads as a sellable top-down one-tool action-adventure, not a primitive debug scene.
