# FOURFOLD ECHOES Decisions

This log is for the private commercial Unity repository.

| ID | Decision | Status | Rationale |
| --- | --- | --- | --- |
| D-001 | Keep the commercial Unity project private. | Accepted | The public repository exists mainly to demonstrate Veripsa Core coordination. Runtime implementation, visual polish, audio direction, asset pipeline, and Steam-facing product work should not reveal avoidable commercial detail. |
| D-002 | Unity 6.3 LTS is the current runtime target. | Accepted | The game now needs fixed-angle 3D feel, animation, lighting, input, controller, build, and Steam-readiness evidence. Unity is the right spike path for that evidence. |
| D-003 | Gate A proved a technical harness, not the next product slice. | Superseded by D-009 and D-012 | Movement, attack, dodge, phase interaction, enemy feedback, gate claim, camera, and placeholder audio were useful evidence, but the resulting room is no longer product direction. |
| D-004 | Prefer Unity's official MCP bridge and reject third-party MCP by default. | Accepted | Unity MCP can expose high-trust editor automation to an AI client. Use Unity's official `com.unity.ai.assistant` MCP path first. Do not install third-party bridges or enable external clients until localhost binding, client approval behavior, generated config, and prompt-injection risk are reviewed. Runtime evidence still requires running the game. |
| D-005 | Keep third-party and generated assets out until provenance is recorded. | Accepted | Steam sale requires asset rights discipline. Gate A uses Unity primitives and procedural placeholder tones only. |
| D-006 | Use CLI/file-contract as the stable Unity gate and MCP only as an acceleration layer. | Accepted | PR and CI evidence must not depend on an interactive editor bridge. MCP may speed up local editor inspection and `Tools/FOURFOLD/...` execution, but the same operation must remain reachable through `-batchmode -executeMethod` and file artifacts. |
| D-007 | Do not claim branch-protection-enforced required checks on the private repo yet. | Accepted | `gh api repos/RollNuts/fourfold-echoes/branches/main/protection` currently returns HTTP 403 requiring GitHub Pro or public visibility. Until that changes, Veripsa/CI can inform merges but cannot be enforced as GitHub required checks on this private repo. |
| D-008 | Reset v1 product scope to single-player premium-indie 3D action adventure. | Accepted | The repo currently contains prototype Gate A evidence, not a sellable product identity. v1 must be independently shippable without online services, co-op, matchmaking, or custom backend cost. D-020 narrows the genre further to compact top-down classic action-adventure. |
| D-009 | Classify Gate A as prototype infrastructure, not canonical gameplay. | Accepted | Gate A proves Unity generation/build/control basics but does not communicate the final product hook or Steam screenshot quality. It can remain as a harness only while D-020 product work moves to one-tool compact top-down action-adventure evidence. |
| D-010 | Remove co-op-ready from v1 canonical scope. | Accepted | Co-op appears as README-level aspiration only; no runtime networking exists. Single-player must be complete first. Future local or invite-only co-op can be revisited after v1 identity is proven. |
| D-011 | Keep graphics and audio standards ahead of feature expansion. | Accepted | The project cannot defer visual/audio identity until late. Product, art, audio, legal, QA, and validation documents now define acceptance bars before more gameplay expansion. |
| D-012 | Do not generate legacy Gate A during normal validation. | Accepted | Normal gates should not recreate the rejected altar/gate prototype or keep pulling agents toward it. Legacy Gate A remains available only with `FOURFOLD_INCLUDE_LEGACY_GATE_A=1` when explicit harness evidence is needed. |
| D-013 | Do not map current product build scripts to legacy Gate A. | Accepted | After the reset, `Scripts/Build/build_current.sh` must fail until a canonical product scene exists. Old Gate A builds stay available only through explicit legacy scripts. |
| D-014 | Make `ProductReviewSandbox` the first generated product-review scene. | Superseded by D-020 | `Assets/Scenes/ProductReviewSandbox.unity` remains useful historical evidence for generation, capture, validation, and build paths, but it is not the current product scene and must not guide future content. |
| D-015 | Canonical v1 is a compact single-player world-state adventure, not an extraction run. | Superseded by D-020 | This decision correctly rejected extraction pressure and recurring server cost, but the world-state/Echo Phase product hook is no longer canonical. |
| D-016 | Treat PS5 as port-ready architecture, not an immediate build promise. | Accepted | PS5 quality means large-screen art direction, controller-first UX, asset/performance budgets, and platform abstraction. Actual PS5 packaging requires partner access, Unity modules, hardware, and certification work that are outside the current evidence gate. |
| D-017 | AI and procedural generation are allowed as production tools, not unreviewed shipped content. | Accepted | Codex, Blender Python, and procedural scripts can accelerate asset factories and evidence. Final player-facing art, audio, text, and store media still need human quality control, provenance, and commercial-use clearance. |
| D-018 | Supersede the four-aspect world-state product direction with a classic compact 3D action adventure. | Superseded by D-019 | This decision remains useful for rejecting extraction pressure, co-op-ready scope, MMO expansion, complex class builds, and loot treadmill framing. It is superseded where it rejects the four-phase world-state hook. |
| D-019 | Restore the four Echo Phases as the canonical product hook for a compact open-world adventure. | Superseded by D-020 | This direction was useful for rejecting co-op, extraction, MMO, live-service, and class-build bloat, but it still expanded the product toward open-world and multiple world-state systems. |
| D-020 | Reset the canonical product to one-tool compact top-down action-adventure. | Accepted | The latest target is stricter: single-player, Steam-first, no open world, no Echo Phase world-state system, no inventory/crafting/quest/social systems, and no system expansion. The MVP is capped at 1 hub, 3 regions, 4 bosses, and 1 exploration tool. Depth must come from repeated mastery of that one tool, room design, enemy reads, shortcuts, boss patterns, art, and audio. |
