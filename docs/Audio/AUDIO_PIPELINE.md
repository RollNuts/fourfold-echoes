# Audio Pipeline

## Workflow

1. Register required cue in `docs/Audio/ASSET_REGISTER.csv`.
2. Confirm commercial-use rights in `docs/Legal/LICENSES.md`.
3. Keep source/master files outside runtime import until approved.
4. Normalize and trim using Audacity or scriptable tooling.
5. Import into Unity with correct load type.
6. Validate loop, clipping, peak level, and category.
7. Capture an audio inventory report.

Current scripted outputs:

- `artifacts/Reports/audio-inventory.json`
- `artifacts/Reports/audio-inventory.md`

These files are generated by `Scripts/Validation/write_market_reports.mjs` from
the audio register and Unity validation metrics. They make missing final SFX,
music, license, and import status visible during every validation run.

Prototype readability SFX can be regenerated with:

```sh
node tools/AudioPipeline/generate_prototype_sfx.mjs
```

The generated files are repository-authored procedural placeholders. They must
remain marked `prototype_only` or `visual_review_only` until replaced or approved
through the normal acceptance process.

## Recommended Free Tools

- Audacity for editing and inspection.
- LMMS for simple composition blockouts.
- ffmpeg for conversion and batch normalization when installed.

## Final Acceptance

No final audio cue may be accepted if:

- source/license is unknown
- loop clicks
- clipping is audible
- it masks boss tells or UI confirmation
- it exists only as an unedited AI generation

## Mix Targets

These are production targets, not proof that current placeholder assets are
accepted.

| Mix Area | Target |
| --- | --- |
| Master peak | below -1 dBTP for release captures |
| Gameplay loudness reference | around -16 LUFS integrated for trailer/gameplay capture checks |
| UI cue peak | audible but below combat hit confirms |
| Enemy tell priority | never masked by BGM or ambience |
| Tool success priority | briefly ducks ambience/music if needed |
| Reward stinger | short, resolved, never blocks control |
| Sudden loudness | avoid jump scares; keep important cues readable, not painful |

Validation must include at least one gameplay capture where tool pulse, tool
success, enemy tell, hit confirm, damage, shortcut open, reward pickup, and UI
confirm can be heard in context.
